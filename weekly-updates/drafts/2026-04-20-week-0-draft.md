# Week 0: The baseline — a UX designer tries to ship something
*Week of April 20 – April 26, 2026*

The foundation post. Where I started, what exists on the canvas today, and what I'm actually trying to figure out by doing this in public.

---

## Where I was at the start of the week

I'm Erik. I'm a UX designer and UI artist by trade — the person who sketches the thing, not the person who usually builds it. I can read code, I can reason about React Flow and component trees, and over the years I've absorbed enough to be dangerous. I am not a trained engineer. Before this project, my definition of "shipping" meant handing a polished Figma file to someone else who would turn it into software.

I'm trying to change that. Not by learning to code the traditional way — I'm 20 years too late for a bootcamp — but by learning to collaborate with an AI the way I used to collaborate with engineers. Which is to say: I bring the product judgment, the UX instincts, and the "no, that's wrong, feel again" reflex; the model brings the syntax, the architecture vocabulary, and the discipline of actually finishing a function.

**The project.** *MasterMind: Story Builder.* A visual, interactive continuity database for Dungeon Masters. Think an investigator's case board — characters, locations, items, factions, story threads — as connected cards on an infinite canvas, backed by a real database so your world doesn't evaporate between sessions. I'm building it for one user first: me, running a paused D&D campaign for my family. If it's useful for us, it has a chance at being useful for other long-campaign DMs.

**Project state at kickoff (April 14).** Empty repo. A design document I'd written ahead of time. A hunch that React Flow could be the canvas. An installed-but-unwired Firebase dependency that I'd copied from some prior experiment and never actually used. The first commit — "Initial canvas with node cards, hover/select states, spacebar pan" — went in on a Tuesday evening.

## What I tried and dropped

**Firebase → Supabase.** Firebase was installed from a prior project and I was going to just keep it. When it came time to actually design the schema, the graph-shaped relationships (`campaigns → nodes → sections`, `connections` pointing at two `nodes`) made Firestore's denormalized document model feel wrong. I also knew Sprint 5 would want vector embeddings for an AI copilot and Supabase ships pgvector. Swapped before I wrote a single Firebase call. ADR-0001 captures the reasoning.

**Lock / unlock cards.** In the design doc I'd specified a lock state for "freeze this card so I don't accidentally drag it." I cut it. I realized while building Sprint 1 that it wasn't going to carry its weight — I don't actually accidentally drag cards, and adding a `locked` column felt like paying for a problem I didn't have. It lives as an in-memory flag in case I change my mind; nothing persists it.

**Duplicate-with-connections.** Also cut. Plain duplicate is enough. I'll probably never miss it.

**The first card-header icon implementation.** The type icon on each card's header was supposed to hide itself when the title got long at extreme zoom-out. My first version made the hide decision depend on the measured `avatarSize`, which itself depended on whether the icon was hidden. At certain zoom levels it would flicker on-off-on-off at 60fps. I spent an embarrassing amount of time debugging that before I realized I'd hand-wired a feedback loop. Rewrote it as a deterministic `useMemo` that simulates the converged layout using canvas `measureText` — no runtime dependency on the thing it's deciding about. The file note in `CampaignNode.jsx` says *"Do not make `iconHidden` depend on runtime-measured `avatarSize`; that path caused oscillation at certain zooms."* That comment is for future-me.

**Committing every day.** I thought I would. I didn't. There's a four-day gap (April 17–20) where I worked and didn't commit a single thing, because I kept telling myself I wasn't *done*. I'm going to try to commit more aggressively going forward even when the work is ugly, because un-committed work is basically undocumented work.

## What I learned

- **AI-era building rewards decisiveness about what you're *not* going to do.** The Sigil post-mortem hit me harder than I expected. Half the weight of this project so far has been saying no — no VTT, no rulebook, no player marketplace, no all-in-one launch. Claude can generate features almost free; the bottleneck is judgment. My job is mostly saying what *not* to build.

- **Schema is expensive. UI is cheap.** ADR-0002. I made `node_sections` a separate child table on day one, even though the Sprint 1 UI still renders four fixed sections. When Sprint 3 adds reorder/add/remove, there's no migration. This is the only place I forced myself to design ahead of the requirement, because it's the one place where being wrong is painful.

- **Stating the constraint beats describing the solution when I'm working with Claude.** The icon-flicker bug didn't get fixed when I said "the icon keeps flickering" — it got fixed when I said "the decision to hide the icon must not depend on any value that depends on whether the icon is hidden." The second sentence is a constraint. The first is a symptom. AI collaboration rewards constraints.

- **My instinct to polish before committing is a bug, not a feature.** See the four-day gap above. A messy commit with a note that says "WIP, icon math still wrong" would have been more useful to me than four days of silence.

- **Optimistic UI is worth the simplicity tax.** ADR-0003. Every handler updates React state first, fires the Supabase write in the background, `.catch(console.error)`s any failures. No spinners, no loading states, no rollback. For a solo tool this is correct. When it's not, I'll know.

- **Getting the stack sync'd with reality is part of the work, not overhead.** I spent the last hour of Sprint 1 deleting dead code (`seedStrahd.js`, `nodeTypes.js`, the Lucide package, the Firebase package). It doesn't feel productive. But the CLAUDE.md and design-document I kept current today are the ones Claude will read next week, and getting those wrong compounds.

## Where I am now

MasterMind has auth. It has campaigns. It has cards with the five built-in types, colored headers, Phosphor icons, the full edit modal, image lightbox, drag-to-reorder bullets, freestanding rich-text annotations, floating edges with dots on the borders, marquee selection, zoom-compensated titles, dynamic icon hiding that doesn't flicker anymore. Everything persists. I can close the browser and come back and the Strahd campaign I seeded is still there.

What it can't do yet: real-time cross-tab sync, undo/redo, reorderable card sections, search, drag-to-connect, relationship labels on edges, anything AI. All on the roadmap, none of it in the product.

I'm not deployed to Vercel yet. The app runs on localhost.
